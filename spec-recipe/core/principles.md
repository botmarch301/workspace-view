# Spec-Recipe Core Principles
> **ID**: CORE-PRC-001
> **Version**: 0.9.0
> **Status**: Draft
> **Last Updated**: 2026-03-19

Spec-Recipe 프레임워크의 도구 독립적 핵심 원칙을 정의한다.

---

## 1. Spec-First

스펙이 코드의 단일 진실 공급원(Single Source of Truth)이다.

- 코드 구현에 선행하여 스펙을 작성한다
- 스펙에 없는 기능을 구현하지 않는다
- 코드에서 스펙 오류를 발견하면, 스펙을 먼저 수정하고 코드를 수정한다

## 2. Machine-Actionable Specification (MAS)

스펙은 인간과 기계가 모두 읽고 검증할 수 있어야 한다.

- 구조화된 마크다운으로 작성한다 (헤더, 리스트, 테이블)
- 기계가 파싱 가능한 메타데이터 헤더를 포함한다
- 모호한 표현("적절히", "필요 시")을 금지하고 구체적 기준을 명시한다

## 3. Separation of Concerns (SoC)

문서의 역할을 명확히 분리한다.

| 역할 | 질문 | 위치 |
|---|---|---|
| Context | 무엇을 정의하는가? | 0-context/ |
| Requirements | 왜 필요한가? | 1-requirements/ |
| Design | 어떻게 구현하는가? | 2-design/ |
| Plan | 언제, 어떤 순서로? | 3-plan/ |
| Manual | 어떤 규칙을 따르는가? | 4-manual/ |

## 4. Traceability

요구사항에서 코드, 코드에서 테스트까지 양방향 추적이 가능해야 한다.

- 모든 추적 가능한 항목은 고유 ID를 갖는다
- 코드에 `Implements: {SPEC-ID}` 태그를 삽입한다
- 테스트에 `Covers: {SPEC-ID}` 태그를 삽입한다
- 추적성 정보는 외부 문서가 아닌 코드 자체에 기재한다

### 4.1 추적성 태그 문법

태그는 해당 언어의 단일행 주석 문법을 사용하되, `Implements:` 또는 `Covers:` 키워드를 반드시 포함한다.

| 언어 | 구현 태그 | 테스트 태그 |
|---|---|---|
| Java, C, JS, TS, Go, Rust | `// Implements: FR-001` | `// Covers: FR-001` |
| Python, Ruby, Shell | `# Implements: FR-001` | `# Covers: FR-001` |
| HTML, XML | `<!-- Implements: FR-001 -->` | `<!-- Covers: FR-001 -->` |
| SQL | `-- Implements: FR-001` | `-- Covers: FR-001` |
| CSS | `/* Implements: FR-001 */` | `/* Covers: FR-001 */` |

검증 스크립트는 `Implements:` 키워드를 언어 무관하게 탐지한다 (정규식: `Implements:\s*[A-Z]+-\d+`).

### 4.2 테스트 의무화

- 모든 기능 요구사항(FR)은 `Covers:` 태그가 있는 테스트를 최소 1개 가져야 한다
- 테스트는 요구사항의 수용 시나리오(Given/When/Then)에서 도출한다
- 비기능 요구사항(NFR)은 검증 방법을 스펙에 반드시 명시한다
- 테스트 커버리지가 기준에 미달하면, Gap 분석 후 보완을 완료해야 Feature를 종료할 수 있다

### 4.3 비기능 요구사항(NFR) 추적

NFR은 크로스커팅 특성상 단일 파일에 귀속시키기 어렵다. 다음 규칙을 따른다:

- NFR을 가장 직접적으로 구현하는 인프라/미들웨어 코드에 태깅한다
- 예: 성능 요구사항 → 캐시 설정 파일, 보안 요구사항 → 인증 미들웨어
- 여러 파일에 걸치는 NFR은 가장 대표적인 1개 파일에 태깅하고, 추적성 매트릭스에 관련 파일 목록을 기재한다
- 스텁 단계에서 NFR 태깅이 어려운 경우, 추적성 매트릭스에 "Pending Implementation" 상태로 기록한다

## 5. Glossary-First

용어를 먼저 정의하고 개발을 시작한다.

- 프로젝트 시작 시 `0-context/glossary.md`를 가장 먼저 작성한다
- 모든 문서와 코드는 용어사전에 정의된 용어를 사용한다
- Enum과 공통 상수는 스펙에서 선행 정의한다 (Enum-First)

## 6. Drift Prevention

스펙과 코드의 불일치(Drift)를 구조적으로 방지한다.

### 6.1 표면적 최소화
설계 문서에서 변동 가능한 구현 세부사항(파일명, 네임스페이스, 수량 등)을 제거하여, 코드 변경 시 문서가 틀려질 수 있는 표면적 자체를 줄인다.

| 제거 대상 | 대체 방법 |
|---|---|
| 파일명 목록 | 구성 원칙만 기술 |
| 네임스페이스 명시 | 패턴/규칙만 기술 |
| 항목 수량 | ID 범위로 정의 |
| 폴더 구조 수동 기재 | 원칙만 기재 |

**예외**: 태스크 계획(Task Plan) 문서는 구체적인 파일 경로를 포함할 수 있다. 태스크는 즉시 실행 가능한 지침이므로 구체성이 필수적이며, 태스크 완료 후 폐기되는 일회성 문서이므로 장기적 drift 위험이 낮다.

### 6.2 변경 연동 트리거
다음 항목이 변경되면 관련 스펙 문서를 반드시 함께 갱신한다:
1. API 경로 (URL, HTTP Method)
2. Request/Response 필드명 또는 타입
3. Enum/Constant 값의 추가, 삭제, 이름 변경
4. DB 컬럼 추가, 삭제, 타입 변경
5. 인증/인가 정책
6. 공개 인터페이스/메서드 시그니처

### 6.3 Drift 원인 분포 (참고)
| 유형 | 비율 | 방지 수단 |
|---|---|---|
| 코드 변경 후 Spec 미갱신 | ~70% | 표면적 감소 + 세션 시작 확인 |
| ADR 결정 후 Spec 미반영 | ~15% | 변경 연동 트리거 |
| 초기 작성 실수 | ~15% | 검증 체크리스트 |

## 7. Verification Separation

검증의 독립성과 실효성을 보장한다.

- **구현 주체와 검증 주체를 분리한다**: 같은 세션/에이전트가 구현과 검증을 동시에 수행하지 않는다
- **기계적 검증을 우선한다**: AI 판단이 아닌 스크립트(패턴매칭, AST 파싱)로 PASS/FAIL을 판정한다
- **의미적 검증은 분리 환경에서 수행한다**: 스펙만 읽은 검증자가 코드를 독립적으로 리뷰한다

## 8. Phase Separation

4개 작업 영역(선언, 설계, 개발, 테스트)을 분리하여, 각 영역의 독립성을 보장한다. (Ref: ADR-0007)

| 영역 | 범위 | 경로 그룹 |
|------|------|----------|
| Specification (선언) | 요구사항, constraint, 예외, ADR | `.spec-recipe/constraints/`, `spec/0-context/`, `spec/1-requirements/`, `decisions/` |
| Design (설계) | 아키텍처, API/스키마, 기술 결정 | `spec/2-design/`, `spec/3-plan/` |
| Development (개발) | 소스 코드, 설정, 빌드 | `src/`, `lib/`, `app/`, `config/` |
| Testing (테스트) | 테스트 코드, 커버리지, 뮤테이션 | `tests/`, `test/`, `**/*_test.*`, `**/*_spec.*` |

### 8.1 분리 규칙

- 동일 커밋에 2개 이상 영역의 파일 변경을 포함하지 않는다 (pre-commit에서 기계적 차단)
- 커밋 메시지에 영역 태그를 필수로 포함한다: `[spec]`, `[design]`, `[dev]`, `[test]`
- AI agent 환경에서는 영역별 세션 분리를 권고한다

### 8.2 분리 근거

- 선언과 설계의 분리: 설계 편의에 맞춰 요구사항을 왜곡하는 것을 방지
- 설계와 개발의 분리: 구현 편의에 맞춰 설계를 임의 변경하는 것을 방지
- 개발과 테스트의 분리: 구현자가 자기 코드에 맞춰 테스트를 왜곡하는 것을 방지
- 선언과 개발의 분리: 개발 중 룰을 임의 변경하는 것을 방지

### 8.3 Verification Separation(원칙 7)과의 관계

원칙 7은 **"누가"의 분리** (구현 주체 ≠ 검증 주체). 원칙 8은 **"언제/무엇을"의 분리** (작업 영역의 독립성). 둘은 상호 보완적이다.

### 8.4 예외

- `spec-recipe init`에 의한 최초 프로젝트 초기화는 여러 영역의 파일을 동시에 생성하므로 예외
- README, CHANGELOG 등 비규칙/비코드 문서는 어느 영역과도 동시 커밋 가능
