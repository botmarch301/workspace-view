# ADR-0003: Invariant 예외 정책

> **ID**: ADR-0003
> **Status**: Accepted
> **Date**: 2026-03-28
> **Deciders**: Mingi
> **Implies**: v0.9.0 제약 체계, CORE-PRC-001 (Verification Separation)

## 1. 맥락 및 문제 정의

Invariant는 절대 위반 불가한 최상위 제약이다. 그러나 현실에서 예외가 필요한 경우가 존재한다 (보안 테스트 코드 등).

**핵심 제약 조건:**
- 예외는 제한적이어야 한다. 전체에 대한 예외가 필요하면 룰 자체를 바꿔야 한다.
- 예외의 적용 대상은 제한적이어야 하고, 사유는 명시적이어야 한다.
- 예외 대상과 예외 사항은 룰에서 명시적으로 관리하며, 임의 수정/삭제를 금지한다.
- **스펙/룰 선언 작업과 설계/개발/테스트 작업은 반드시 분리**되어야 한다. 개발 중에 룰이나 예외를 임의로 만들 수 없다.

## 2. 고려된 대안

### 2.1 대안 1: 완전 금지
- **Pros**: 절대성 보장
- **Cons**: false positive 시 탈출구 없음 → 프레임워크 회피

### 2.2 대안 2: 자유 비활성화
- **Pros**: 유연
- **Cons**: invariant 무력화

### 2.3 대안 3: 제한적 예외 선언제 (룰 레벨 관리, 작업 분리 강제)
- **Pros**: 예외 가능하되, 룰의 일부로 관리됨. 개발 작업에서 예외를 만들 수 없음.
- **Cons**: 프로세스 오버헤드. 하지만 invariant의 성격상 당연한 비용.

## 3. 결정

**대안 3: 제한적 예외 선언제**를 선택한다.

### 3.1 핵심 원칙

1. **예외는 룰의 일부다.** 예외는 constraint 정의와 같은 레벨에서 관리한다. 개발/테스트 작업 중에 예외를 생성/수정/삭제할 수 없다.

2. **적용 대상을 최소화한다.** 예외는 반드시 구체적 파일 + 라인 범위로 한정한다. 디렉토리 전체나 와일드카드 범위의 예외는 허용하지 않는다. 넓은 범위의 예외가 필요하면 룰 자체를 재검토한다.

3. **사유를 명시한다.** 왜 이 예외가 필요한지, 어떤 상황에서만 유효한지를 기록한다.

4. **임의 수정/삭제를 금지한다.** 예외의 변경은 ADR 또는 이슈 레지스트리를 통해서만 가능하다. phase gate에서 리뷰 대상이다.

5. **만료를 강제한다.** 영구 예외는 없다. 만료 시 자동 HALT 복원.

### 3.2 작업 분리 강제

```
[룰/스펙 선언 작업]              [개발/테스트 작업]
  │                                │
  ├── constraint 정의/수정          ├── 코드 작성
  ├── 예외 선언/수정               ├── 테스트 작성
  ├── ADR 작성                     ├── 리팩토링
  └── build (agent 규칙 생성)      └── 커밋
       │                                │
       └── 반드시 별도 커밋/세션 ────────┘
```

- 개발 세션에서 "이 파일에 예외를 추가해주세요"는 **불가**
- 예외가 필요하면 개발을 중단하고, 룰 선언 작업으로 전환하여 예외를 먼저 등록한 후 개발 재개
- pre-commit에서 constraint/ 변경과 src/ 변경이 같은 커밋에 포함되면 차단

### 3.3 예외 정의 형식

```yaml
# .spec-recipe/constraints/exceptions/exc-001.yml
id: EXC-001
target_constraint: INV-SEC-001
scope:
  files:
    - "tests/security/test_injection.py"
  lines: [42, 43, 44]           # 필수. 파일 전체 예외 불가.
reason: "SQL 인젝션 탐지 테스트를 위한 의도적 취약 패턴"
expires: 2026-06-30             # 필수. 최대 90일.
approved_by: "Mingi"
created: 2026-03-28
related_adr: ADR-0003           # 또는 관련 이슈 ID
```

### 3.4 예외 범위 제한 규칙

| 허용 | 불허 |
|------|------|
| 특정 파일 + 특정 라인 | 디렉토리 전체 (`tests/**`) |
| 1~2개 파일 | 와일드카드 (`*.py`) |
| 명시적 사유 | "필요해서" 등 비구체적 사유 |
| 만료일 90일 이내 | 만료일 없음 또는 1년 이상 |

넓은 범위의 예외가 필요한 경우 → **룰 자체를 재검토**하여 scope 조정 또는 detection 패턴 수정.

### 3.5 검증 메커니즘

- 예외 파일의 유효성 검증 (필수 필드, 범위 제한, 만료일)은 pre-commit에서 자동 수행
- 만료된 예외가 있으면 HALT
- constraint/ 변경과 src/ 변경의 동시 커밋 차단
- phase gate에서 활성 예외 목록 전체 리뷰 필수

## 4. 영향

- **Positive**: 예외가 룰의 일부로 관리됨, 작업 분리 강제, 임의 조작 불가, 감사 추적 가능
- **Negative**: 예외 등록 절차의 오버헤드 (하지만 invariant 예외는 본래 드문 상황)
- **Follow-up**:
  - constraints/exceptions/ 디렉토리 및 스키마 정의
  - pre-commit: 예외 유효성 + 만료 검사 + constraint/src 동시 변경 차단
  - 이슈 레지스트리 연동
  - processes.md에 "룰 선언 작업과 개발 작업의 분리" 원칙 추가
