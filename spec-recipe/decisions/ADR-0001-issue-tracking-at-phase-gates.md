# ADR-0001: 프로세스 전환 시점(Phase Gate)에서의 이슈 추적 강제

> **ID**: ADR-0001
> **Status**: Accepted
> **Date**: 2026-03-28
> **Deciders**: Mingi
> **Implies**: CORE-PRC-002 (processes.md), CORE-VRF-001 (verification.md)

## 1. 맥락 및 문제 정의

spec-recipe는 요구사항 → 설계 → 구현 → 테스트의 프로세스를 정의하고 있다. 그러나 각 단계를 전환할 때 발생하는 이슈(불일치, 누락, 중복, 오기록)를 체계적으로 추적하고 관리하는 메커니즘이 없다.

**문제 상황 예시:**
- 포워드: 요구사항 → 설계 시, 요구사항에 모호한 항목이 발견됨 → 기록 없이 설계자가 임의 해석
- 포워드: 설계 → 구현 시, 설계에 없는 기능을 구현자가 추가 → Hidden Feature 발생
- 백워드: 테스트 → 구현 시, 테스트 실패가 설계 결함인지 구현 버그인지 분류 안 됨
- 백워드: 구현 → 요구사항 시, 기존 코드 분석 중 요구사항에 없는 기능 발견 → 스펙 보완 필요

이런 이슈들이 기록되지 않으면:
1. 같은 문제를 다른 세션에서 반복 발견
2. 교정 시 근거 없이 임의 판단
3. 이슈 해결 여부 추적 불가

**제약 조건:**
- AI agent가 프로세스를 실행하므로, agent가 이슈를 자동으로 기록/확인할 수 있어야 한다
- 포워드 엔지니어링과 백워드(리버스) 엔지니어링 양쪽에서 모두 작동해야 한다

## 2. 고려된 대안

### 2.1 대안 1: 이슈를 별도 파일로 관리 (Issues Registry)

프로젝트에 `.spec-recipe/issues/` 디렉토리를 두고, 이슈를 개별 파일 또는 단일 레지스트리 파일로 관리한다. 프로세스 전환 시(phase gate) 이슈 레지스트리 확인을 강제한다.

- **Pros**: 이슈가 독립적으로 추적 가능, 상태 관리 명확, agent가 파일 읽기/쓰기로 처리
- **Cons**: 파일이 많아질 수 있음, 프로세스 문서와 분리되어 참조 비용

### 2.2 대안 2: 이슈를 해당 프로세스 산출물에 인라인 기록

각 스펙/설계/코드 문서 안에 이슈 섹션을 추가한다.

- **Pros**: 이슈가 맥락과 함께 존재, 별도 관리 불필요
- **Cons**: 문서가 비대해짐, 이슈 상태 추적이 어려움, 전체 이슈 조회 불가

### 2.3 대안 3: 이슈 레지스트리 + Phase Gate 체크리스트 결합

2.1의 레지스트리에 프로세스 전환 시점별 체크리스트를 결합한다. 각 phase gate에서 "미해결 이슈 0건" 확인을 통과 조건으로 추가한다.

- **Pros**: 체계적 추적 + 강제 게이트, 포워드/백워드 모두 커버
- **Cons**: 관리 오버헤드, 하지만 자동화로 완화 가능

## 3. 결정

**대안 3: 이슈 레지스트리 + Phase Gate 체크리스트**를 선택한다.

### 3.1 결정의 근거

1. 이슈가 프로세스 단계와 독립적으로 존재해야 **교차 참조**가 가능하다 (요구사항에서 발견된 이슈가 설계에 영향을 미치는 경우)
2. Phase gate에서의 강제 확인은 spec-recipe의 "기계적 강제 우선" 원칙과 일치한다
3. 단일 레지스트리 파일이면 agent가 한 번 읽기로 전체 이슈를 파악할 수 있다
4. 포워드/백워드 양방향의 이슈를 "발생 맥락(source → target)"으로 분류하면 방향성이 명확해진다

### 3.2 설계

#### 이슈 레지스트리 위치

```
.spec-recipe/
  issues/
    registry.md     # 전체 이슈 목록 (단일 파일, 자동 관리)
    archive/        # 해결된 이슈 아카이브
```

#### 이슈 형식

```markdown
## ISS-001: [이슈 제목]
- **상태**: open | in-progress | resolved | wontfix
- **발견 시점**: [phase] (예: 요구사항→설계)
- **방향**: forward | backward | cross-cutting
- **발생 맥락**: [어떤 작업 중 발견되었는지]
- **영향 범위**: [영향받는 문서/코드 목록]
- **설명**: [구체적 문제 내용]
- **조치**: [해결 방법 또는 결정사항]
- **해결일**: [해결 시 기재]
```

#### Phase Gate 체크리스트

프로세스 전환 시 다음을 **기계적으로** 확인한다:

| Phase Gate | 전환 | 확인 항목 |
|------------|------|----------|
| PG-1 | 요구사항 → 설계 | 요구사항 관련 open 이슈 0건, 모호한 요구사항 없음, 용어집 완성 |
| PG-2 | 설계 → 구현 | 설계 관련 open 이슈 0건, 추적성 매핑 완료, API/스키마 정의 완료 |
| PG-3 | 구현 → 테스트 | 구현 관련 open 이슈 0건, 추적성 태그 부여 완료, 정적 분석 PASS |
| PG-4 | 테스트 → 완료 | 전체 open 이슈 0건, 테스트 커버리지 충족, drift 0건 |
| PG-R1 | 코드 → 요구사항 (리버스) | 미문서화 기능(CODE-ONLY) 0건 또는 이슈로 등록 |
| PG-R2 | 테스트 → 설계 (리버스) | 테스트 실패의 원인 분류 완료 (설계 결함 vs 구현 버그) |

**강제 방식**: phase gate 통과 스크립트가 registry.md를 파싱하여 해당 단계의 open 이슈가 0건인지 확인. 0건이 아니면 다음 단계 진행을 차단한다.

## 4. 영향

- **Positive**: 이슈가 누락 없이 추적됨, 프로세스 전환 품질 향상, 세션 간 연속성 확보
- **Negative**: 관리 오버헤드 증가 (자동화로 완화)
- **Follow-up**: 
  - processes.md에 Phase Gate 체크리스트 추가
  - verification.md에 Phase Gate 검증 항목 추가
  - init.py에 issues/ 디렉토리 초기화 추가
  - pre-commit hook에서 phase gate 체크 연동
