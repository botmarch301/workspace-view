# Harness Engineering — AI 코딩 에이전트를 제어하는 기술

> AI 코딩 에이전트가 만든 코드를 신뢰할 수 있으려면, 코드를 만드는 환경 자체를 설계해야 한다.

---

## 하네스 엔지니어링이란

2026년 초, OpenAI가 Codex를 활용한 내부 실험 결과를 공개하면서 "Harness Engineering"이라는 용어가 공식적으로 등장했다. 100만 줄의 코드를 수동 작성 0줄로 만들었다는 실험이다. 엔지니어 3명이 5개월간 1,500개의 PR을 병합했고, 1인당 하루 평균 3.5개의 PR을 처리했다.

핵심 원칙은 간단하다.

> "Humans steer. Agents execute."

사람은 방향을 잡고, 에이전트가 실행한다. 여기서 "방향을 잡는다"는 것이 단순히 프롬프트를 잘 쓰는 것이 아니다. 에이전트가 작업하는 환경 자체를 설계하는 것이다. 이것이 하네스 엔지니어링이다.

전통적인 test harness가 "인간이 작성한 코드가 맞는지 검증하는 환경"이라면, agent harness engineering은 "AI 에이전트가 올바른 코드를 만들 수 있는 환경 전체를 설계하는 것"이다.

| 구분 | Test Harness | Agent Harness Engineering |
|------|-------------|--------------------------|
| 대상 | 인간이 작성한 코드 검증 | AI agent의 코드 생산 환경 전체 |
| 범위 | 테스트 실행, 리포트 | 스펙 전달 → 코드 생성 → 검증 → 피드백 루프 |
| 핵심 질문 | 코드가 맞는가? | agent가 올바른 코드를 만들 수 있는 환경인가? |
| 설계 대상 | 테스트 프레임워크 | 문서 구조, 제약 시스템, 컨텍스트 관리, 자동화 |

---

## 하네스 엔지니어링이 없으면

AI 코딩 에이전트를 아무 환경에 그냥 풀어놓으면 어떤 일이 벌어지는가.

**1. 아키텍처 부식**

에이전트는 "지금 당장 동작하는 코드"를 만드는 데 최적화되어 있다. 프로젝트 전체의 아키텍처 원칙이나 모듈 경계를 지키는 것은 관심 밖이다. 계층 간 의존성이 꼬이고, 추상화가 무너지고, 코드베이스가 점진적으로 썩는다. 인간 개발자도 같은 실수를 하지만, 에이전트는 훨씬 빠른 속도로 같은 실수를 반복한다.

**2. 컨텍스트 분실**

에이전트가 볼 수 없는 정보는 존재하지 않는 것과 같다. Slack에서 합의한 설계 결정, Google Docs에 적은 요구사항, 팀원의 머릿속에 있는 암묵지 — 이것들은 에이전트의 세계에 없다. 결과적으로 에이전트는 이미 결정된 사항을 무시하고, 이미 해결된 문제를 다시 만들어낸다.

**3. 검증 부재**

에이전트가 "완료했습니다"라고 하면 정말 완료된 건가? 테스트가 통과한다고 해서 요구사항을 충족하는 건 아니다. 스펙과 구현 사이의 정합성을 기계적으로 검증하지 않으면, 겉으로는 동작하지만 실제로는 다른 것을 만드는 상황이 생긴다.

**4. 피드백 루프 부재**

에이전트가 실패했을 때 무엇이 잘못됐는지 자동으로 파악하고 재시도하는 구조가 없으면, 결국 인간이 하나하나 확인하고 다시 지시해야 한다. 이러면 에이전트를 쓰는 의미가 반감된다.

**5. 규모 확장 불가**

위 문제들은 코드베이스가 작을 때는 사람이 감당할 수 있다. 하지만 코드가 수천, 수만 줄을 넘기는 순간 사람의 주의력만으로는 통제할 수 없게 된다. OpenAI가 하네스 엔지니어링을 제안한 것도 바로 이 지점이다.

## 하네스 엔지니어링은 이 문제를 어떻게 해결하는가

하네스 엔지니어링의 접근은 "에이전트에게 잘 시키는 것"이 아니라 "에이전트가 잘못하기 어려운 환경을 만드는 것"이다.

### 저장소를 시스템 오브 레코드로

에이전트가 참조할 모든 정보를 저장소 안에 버전 관리한다. 아키텍처 문서, 설계 원칙, 코딩 표준, 실행 계획이 마크다운 파일로 저장소에 존재한다. Slack 스레드도, Google Docs도, 사람의 기억도 아닌 — 저장소가 유일한 진실의 원천이다.

OpenAI는 이를 "AGENTS.md를 백과사전이 아닌 목차로 취급하라"고 표현했다. AGENTS.md는 100줄 내외의 짧은 진입점이고, 실제 지식은 `docs/` 디렉토리에 구조화되어 있다.

### 기계적 강제

인간에게 "이 규칙을 지켜주세요"라고 부탁하듯 에이전트에게도 부탁만 하면 안 된다. 에이전트는 지시를 따를 수도 있고 무시할 수도 있다. 따라서 규칙은 기계적으로 강제한다.

- **아키텍처 경계**: 커스텀 린터가 모듈 간 import 방향을 검사한다. 위반하면 CI가 실패한다.
- **코딩 표준**: pre-commit hook이 네이밍 컨벤션, 파일 크기 제한, 구조화된 로깅을 강제한다.
- **품질 게이트**: 테스트 통과, 커버리지 임계값, 정적 분석 통과가 PR 병합 조건이다.

에이전트가 만든 코드가 이 게이트를 통과하지 못하면 자동으로 거부된다. 여기에 에이전트의 "판단"이 개입할 여지는 없다.

### 피드백 루프

검증 실패 → 에이전트에게 실패 원인 전달 → 에이전트가 수정 → 재검증. 이 루프가 자동으로 돌아야 한다. OpenAI의 경우 에이전트가 자기 변경사항을 스스로 리뷰하고, 다른 에이전트에게도 리뷰를 요청하며, 모든 리뷰어가 만족할 때까지 반복하는 루프를 구축했다.

### 점진적 공개

에이전트에게 한꺼번에 모든 정보를 주지 않는다. 작은 진입점(AGENTS.md)에서 시작해서, 필요한 시점에 필요한 문서를 참조하도록 안내한다. 컨텍스트 윈도우는 유한하다. 불필요한 정보로 채우면 정작 중요한 정보를 놓친다.

### 격리된 실행 환경

각 작업을 독립된 환경에서 실행한다. git worktree 단위로 앱을 부팅하고, 임시 관찰 스택(로그, 메트릭)을 붙이고, 작업이 끝나면 환경을 정리한다. 한 에이전트의 작업이 다른 에이전트의 작업에 영향을 주지 않는다.

## 하네스 엔지니어링의 주요 개념

### 6개 레이어 모델

하네스 엔지니어링 도구들을 분석하면, 관심사가 6개 레이어로 나뉜다.

```
L5. 평가/측정     리포의 하네스 수준을 점수화하고 추적
L4. 피드백 루프    실패 → 자동 재시도, agent 간 리뷰, 자기 수정
L3. 검증          스펙↔코드↔테스트 일치 검사, 기계적 PASS/FAIL
L2. 제약 주입      표준/규칙을 agent 컨텍스트에 주입
L1. 스펙 관리      요구사항 → 스펙 → 계획 → 태스크 분해
L0. 런타임 하네스   agent 실행 루프, 도구 디스패치, 컨텍스트 관리
```

모든 레이어를 커버하는 도구는 아직 없다. 대부분의 도구는 하나 또는 두 개의 레이어에 특화되어 있다.

### 품질 특성

하네스 엔지니어링의 품질을 판단하는 주요 특성은 다음과 같다.

**결정론(Determinism)**: 같은 입력에 같은 결과를 내는가. `just check` 같은 단일 하네스 커맨드가 있어서, 언제 어디서 실행해도 같은 검증 결과를 내는 것이 이상적이다.

**기계적 강제(Mechanical Enforcement)**: 규칙이 문서에만 존재하는가, 아니면 코드로 강제되는가. 문서만으로는 에이전트가 무시할 수 있다. git hook, 린터, CI 게이트로 강제해야 한다.

**점진적 공개(Progressive Disclosure)**: 에이전트에게 한꺼번에 모든 컨텍스트를 주는가, 아니면 필요한 시점에 필요한 만큼 제공하는가.

**도구 독립성(Tool Agnosticism)**: 특정 에이전트(Claude, GPT, Gemini 등)에 종속되는가, 아니면 어떤 에이전트에서든 작동하는가.

**Brownfield 호환성**: 새 프로젝트(greenfield)에서만 쓸 수 있는가, 기존 프로젝트(brownfield)에도 적용할 수 있는가.

**증거 기반 변경(Evidence-Based Changes)**: 모든 변경이 테스트, 스냅샷, golden diff 같은 증거를 동반하는가.

### 1. Harness Engineering Playbook

| 항목 | 내용 |
|------|------|
| 저장소 | [lipingtababa/harness-engineering-playbook](https://github.com/lipingtababa/harness-engineering-playbook) |
| Stars | 42 |
| 언어 | Shell |
| 핵심 개념 | 팀 환경에서 AI agent와 품질 소프트웨어를 출하하기 위한 실전 프레임워크 |

**작동 방식**

OpenAI의 하네스 엔지니어링 원칙을 팀 환경용 Playbook으로 체계화한 프레임워크다. 결정론적 smoke/test/lint 하네스 커맨드, 엄격한 아키텍처 경계, 엔트로피 제어를 핵심으로 한다.

단일 하네스 커맨드(`just check` 또는 동등한 것)가 모든 게이트를 로컬과 CI에서 동일하게 실행한다. 아키텍처 경계는 Import Linter, grimp 같은 도구로 기계적으로 강제한다. AST 규칙(ast-grep)으로 금지 패턴을 탐지하고 회귀를 방지한다.

**특징**
- "팀 환경"을 명시적으로 고려한 유일한 프레임워크
- 에이전트 1명이 아닌, 여러 에이전트 + 여러 엔지니어가 협업하는 상황 상정
- 커스텀 린트의 에러 메시지를 에이전트 컨텍스트에 주입하는 패턴
- 증거 기반 변경: 모든 PR에 테스트, 스냅샷, golden diff 동반

**장점**
- OpenAI 원칙을 가장 충실하게 Playbook으로 옮김
- 결정론적 단일 커맨드 패턴이 명확
- 팀 규모 확장에 대한 가이드 포함

**단점**
- Shell 스크립트 기반이라 설정 자동화 도구는 아님
- 프레임워크라기보다 문서에 가까움
- 직접 실행해서 결과를 내주는 기능 없음

**레이어**: L2(제약 주입), L3(검증)

### 2. Entrix

| 항목 | 내용 |
|------|------|
| 저장소 | [phodal/entrix](https://github.com/phodal/entrix) |
| Stars | 31 |
| 언어 | Python |
| 핵심 개념 | 품질 규칙과 아키텍처 제약을 실행 가능한 가드레일로 변환 |

**작동 방식**

규칙을 코드화하고, 변경사항에 대해 자동으로 실행하는 도구다. 세 단계의 검증 티어(fast/normal/deep)를 제공하며, 변경 인식(change-aware) 검증으로 diff에 대해서만 가중 점수를 매기고 하드 게이트를 적용한다.

`entrix check --tier fast`로 빠른 검증을, `--tier deep`으로 정밀 검증을 실행한다. `review-trigger`로 위험한 변경을 더 깊은 검증 단계로 라우팅한다. 그래프 기반 영향도 분석, 테스트 반경 추정, 리뷰 컨텍스트 분석을 선택적으로 추가할 수 있다.

CLI 단독 사용과 Claude Code 플러그인 두 가지 설치 경로를 제공한다.

**특징**
- "규칙 → 실행 가능한 가드레일" 변환이 핵심 가치
- 변경 수명주기(lifecycle)에 맞춰 검증 시점을 자동 배치
- 정책으로 hard-stop, warn, 인간 리뷰 에스컬레이션을 자동 라우팅

**장점**
- 기계적 강제를 가장 직접적으로 구현
- tier별 검증으로 빠른 피드백과 정밀 검증을 양립
- PyPI 배포로 설치 간편

**단점**
- Python 3.10+ 필요
- 규칙 정의에 초기 투자 필요
- 스펙 관리 기능 없음 (L1 미지원)

**레이어**: L2(제약 주입), L3(검증), L4(피드백 루프)

### 3. Harness Kit

| 항목 | 내용 |
|------|------|
| 저장소 | [deepklarity/harness-kit](https://github.com/deepklarity/harness-kit) |
| Stars | 29 |
| 언어 | Python |
| 핵심 개념 | AI agent 오케스트레이션 + TDD 우선 실행 + 지식 축적 |

**작동 방식**

스펙을 입력하면 태스크를 DAG(방향성 비순환 그래프)로 분해하고, 다중 에이전트에게 병렬 실행을 위임한다. 칸반 보드 UI로 진행 상황을 추적하고, 각 태스크의 비용, 소요 시간, 에이전트별 증거(proof of work)를 기록한다.

"시스템은 당신이 주는 스펙만큼만 좋다"가 철학이다. 코드가 아니라 스펙에 시간을 쓰라는 것이다.

**특징**
- 시각적 보드 뷰, 스펙 실행 뷰, DAG 뷰, 태스크 상세 뷰 제공
- 비용 인식 위임: 에이전트별 비용 추적
- 각 실행이 다음 실행을 개선하는 복합 성장 패턴
- TDD 우선: 테스트를 먼저 작성하고 구현을 에이전트에게 위임

**장점**
- 시각적 관리 인터페이스가 있는 유일한 도구
- 다중 에이전트 병렬 실행
- 비용 추적으로 경제적 의사결정 지원

**단점**
- 초기 단계(WIP), 많은 부분이 실험적
- 샌드박스 미지원 (계획 중)
- 설정과 운영이 복잡

**레이어**: L0(런타임), L1(스펙 관리)

| 핵심 개념 | CEO처럼 지시하면 AI 에이전트 팀이 실행 |

**작동 방식**

"코딩을 하지 말고, AI 팀을 운영하라"가 슬로건이다. 사용자가 지시("대시보드에 다크 모드 추가")를 내리면, 에이전트들이 접근 방식을 브레인스토밍하고, 가정에 도전하고, 구현하고, 서로의 작업을 리뷰하고, 출하한다.

컨텍스트 엔지니어링과 하네스 엔지니어링을 결합해서, 에이전트 팀이 기관적 지식(institutional knowledge)을 축적하는 컨텍스트 트리를 운영한다. 속도보다 깊이를 추구한다.

**특징**
- 엔지니어링뿐 아니라 마케팅, 운영까지 포괄하는 멀티 에이전트
- 에이전트 간 리뷰, 도전, 피드백 루프
- 컨텍스트 트리 기반 지식 축적

**장점**
- 1인 개발자/소규모 팀에 최적화
- 에이전트 간 협업이 가장 발전된 형태
- npm 패키지로 설치 간편

**단점**
- Alpha 단계
- 대규모 팀에서의 검증 부족
- TypeScript 생태계에 의존

**레이어**: L0(런타임), L1(스펙 관리), L4(피드백 루프)

### 5. Harness Engineering (jrenaldi79)

| 항목 | 내용 |
|------|------|
| 저장소 | [jrenaldi79/harness-engineering](https://github.com/jrenaldi79/harness-engineering) |
| Stars | 47 |
| 언어 | JavaScript |
| 핵심 개념 | 기계적 강제를 통한 코딩 에이전트 제어 |

**작동 방식**

Andrej Karpathy의 "에이전트는 내 지시를 듣지 않는다"라는 문제 인식에서 출발한다. 더 나은 프롬프팅이 답이 아니라, 기계적 강제가 답이라는 입장이다.

CLAUDE.md 템플릿, git hook, pre-commit 설정을 제공하고, 20가지 이상의 베스트 프랙티스를 필드 가이드로 정리한다. 에이전트가 규칙을 "읽고 이해하길 바라는" 대신 git hook이 위반을 물리적으로 차단한다.

**특징**
- "기계적 강제" 철학을 가장 명확하게 실천
- CLAUDE.md 템플릿 + 20+ 베스트 프랙티스 필드 가이드
- Claude Code 플러그인으로 설치 가능
- Copilot, Codex, Cursor 등 다양한 에이전트 지원

**장점**
- 설치 후 즉시 작동하는 실전적 접근
- 기계적 강제 메커니즘이 구체적
- 잘 정리된 베스트 프랙티스 가이드

**단점**
- "강제"에 집중하고 "안내"가 부족
- 스펙 관리, 태스크 분해 기능 없음
- 프레임워크보다 도구 모음에 가까움

**레이어**: L2(제약 주입)

### 6. AI Harness Scorecard (주시)

| 항목 | 내용 |
|------|------|
| 저장소 | [markmishaev76/ai-harness-scorecard](https://github.com/markmishaev76/ai-harness-scorecard) |
| Stars | 14 |
| 언어 | Python |
| 핵심 개념 | 리포지토리의 agent 코드 신뢰도를 정량 평가 |

**작동 방식**

`ai-harness-scorecard assess .` 하나로 현재 리포의 하네스 수준을 A~F 등급으로 평가한다. 5개 카테고리, 31개 체크를 수행한다.

1. **아키텍처 문서화** (20%): 아키텍처 문서, 에이전트 지시, ADR, 모듈 경계 제약
2. **기계적 제약** (25%): CI, 린터, 타입 안전, 의존성 감사
3. **테스트와 안정성** (25%): 테스트, 커버리지, 뮤테이션, 퍼즈
4. **리뷰와 드리프트** (15%): 코드 리뷰 강제, stale 문서 탐지
5. **AI 특화 세이프가드** (15%): AI 사용 규범, 소규모 배치, 에러 처리

DORA 2025 보고서, OpenAI Harness Engineering, SlopCodeBench, Kent Beck의 테스트 원칙을 근거로 한다.

**주시하는 이유**

기존 도구들이 "하네스를 만드는 것"에 집중할 때, 이 도구는 "하네스를 측정하는 것"에 집중한다. "agent가 이 리포에서 신뢰할 수 있는 코드를 만들 수 있는가?"라는 질문에 정량적으로 답한다. CI/CD 통합이 가능해서 지속적으로 하네스 품질을 추적할 수 있다.

**레이어**: L5(평가/측정)

### 7. Reins (주시)

| 항목 | 내용 |
|------|------|
| 저장소 | [WellDunDun/reins](https://github.com/WellDunDun/reins) |
| Stars | 6 |
| 언어 | TypeScript |
| 핵심 개념 | OpenAI 방법론의 CLI 도구화 |

**작동 방식**

"OpenAI가 방법론을 발표했고, 우리는 도구를 만들었다."

4단계 CLI로 리포의 하네스 성숙도를 점진적으로 높인다.

1. `reins scaffold` — 하네스 구조 초기 생성
2. `reins audit` — 현재 리포의 하네스 수준 진단
3. `reins evolve` — 진단 결과를 기반으로 개선
4. `reins doctor` — 하네스 건강 상태 확인

**주시하는 이유**

의존성 0개, Node.js 18+만 있으면 동작한다. 하네스 엔지니어링을 "도입하는 과정"을 단계적으로 안내하는 접근이 실용적이다. scaffold → audit → evolve → doctor라는 생명주기가 명확하다. 기존 프로젝트(brownfield)에도 즉시 적용 가능하다.

**레이어**: L2(제약 주입), L5(평가/측정)

| 언어 | Python |

## 적용 제안

### Greenfield (새 프로젝트)

새 프로젝트를 시작한다면 하네스를 처음부터 설계할 수 있다.

**추천 조합**: Harness Kit(스펙 관리 + 런타임) + Entrix(검증 가드레일) + AI Harness Scorecard(수준 측정)

1. Harness Kit로 스펙을 정의하고 태스크를 분해한다
2. Entrix로 품질 규칙과 아키텍처 제약을 코드화한다
3. 에이전트가 코드를 생성하면 Entrix가 자동 검증한다
4. AI Harness Scorecard로 주기적으로 하네스 수준을 측정한다

초기 투자가 크지만, 코드베이스가 커질수록 복리 효과가 나타난다.

### Brownfield (기존 프로젝트)

기존 프로젝트에 하네스 엔지니어링을 도입한다면 점진적 접근이 필요하다.

**추천 조합**: Reins(점진적 도입) + jrenaldi79(기계적 강제) + AI Harness Scorecard(현황 파악)

1. AI Harness Scorecard로 현재 리포의 하네스 수준을 측정한다
2. Reins의 `scaffold` → `audit` → `evolve` 사이클로 점진적으로 하네스를 구축한다
3. jrenaldi79의 git hook과 pre-commit으로 기계적 강제를 추가한다
4. 주기적으로 Scorecard를 돌려서 개선을 추적한다

한꺼번에 전부 도입하려 하면 실패한다. 가장 중요한 규칙 3~5개부터 기계적으로 강제하고, 점진적으로 늘려가는 것이 현실적이다.

### 규모별 고려사항

**개인/1인 개발자**: gruAI가 적합하다. 에이전트 팀을 운영하듯 작업할 수 있고, 설정 부담이 적다. Playbook 수준의 엄격한 하네스는 오버킬이다.

**소규모 팀 (2~5명)**: Playbook + Entrix 조합이 적합하다. 팀 환경에서의 일관성이 중요해지는 시점이고, Playbook이 이를 직접 다룬다.

**중대규모 팀 (10명+)**: 전체 레이어를 커버하는 조합이 필요하다. L0~L5를 각각 다른 도구로 채우되, CI/CD 파이프라인으로 통합해야 한다. 이 규모에서는 하네스 엔지니어링 자체가 하나의 전문 영역이 된다.

## 현재 생태계의 한계

하네스 엔지니어링은 아직 초기 단계다. 몇 가지 명확한 한계가 있다.

**1. 파편화**: 도구마다 다른 레이어에 특화되어 있고, 도구 간 통합 표준이 없다. Entrix의 검증 결과를 gruAI의 피드백 루프에 자동으로 연결하는 식의 통합은 아직 불가능하다.

**2. 검증의 한계**: 대부분의 "검증"이 정적 분석(린터, 타입 체크, 아키텍처 경계)에 머물러 있다. 스펙과 구현의 의미적 정합성을 기계적으로 검증하는 도구는 아직 없다.

**3. 성숙도**: 대부분의 도구가 2026년 1분기에 등장했다. 대규모 실전 검증이 이루어진 도구는 거의 없다. OpenAI의 내부 실험 외에 공개된 대규모 사례가 부족하다.

**4. 규칙 형식 파편화**: CLAUDE.md, .cursor/rules, AGENTS.md, .kiro/steering 등 에이전트마다 규칙 형식이 다르다. AI-Rule-Spec 같은 통합 표준 시도가 있지만 아직 초기다.

그럼에도, 하네스 엔지니어링은 AI 코딩 에이전트를 실전에서 쓰려면 반드시 필요한 분야다. "에이전트가 더 똑똑해지면 해결될 문제"가 아니다. 에이전트가 똑똑해질수록, 그 능력을 올바른 방향으로 쓰게 하는 하네스의 중요성은 더 커진다.
